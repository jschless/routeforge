## Concept walkthrough

### The problem: keeping a distributed topology database consistent

OSPF is a link-state protocol. Every router in an OSPF area maintains its own copy of the Link State Database (LSDB) — a collection of Link State Advertisements (LSAs) that together describe the complete topology of the area. Because LSAs are generated by individual routers and flooded across the network, every router eventually receives every LSA. The LSDB is the shared ground truth from which each router independently runs Dijkstra's algorithm (SPF) to compute its own routing table.

Keeping this database consistent across dozens of routers in a dynamic network requires solving two related problems: reliably distributing new and updated LSAs (flooding), and eventually removing stale LSAs from routers that no longer have fresh information about them (aging).

### Key terms

- **LSA (Link State Advertisement)**: A structured record that describes a piece of the network topology — for example, a router's active links and their costs, or the set of routers attached to a network prefix. Each LSA is identified by a `(lsa_id, advertising_router)` pair.
- **LSDB (Link State Database)**: The collection of all LSA records known to a router. Every router in an area converges to an identical LSDB under normal operation.
- **Sequence number**: A monotonically increasing integer attached to each LSA instance. When a router refreshes or updates an LSA it originates, it increments the sequence number. A receiver that already holds a copy compares sequence numbers to decide whether the incoming LSA is newer, a duplicate, or stale.
- **Age**: An integer counter (in seconds) that starts at 0 when an LSA is first generated and increments as the LSA propagates and ages in the LSDB. When age reaches `max_age` (typically 3600 seconds, i.e. one hour), the LSA is considered expired and must be removed.
- **MaxAge**: The ceiling value for LSA age. An LSA that reaches MaxAge is flooded one final time with age set to MaxAge so neighbors can also remove it, then purged from the local LSDB.
- **LSA refresh**: Before an LSA reaches MaxAge, the originating router re-originates it with an incremented sequence number and age reset to 0. This keeps the LSA alive in the network indefinitely as long as the originating router is present.

### How LSA flooding works

When a router receives a new or updated LSA, it:

1. Checks whether the incoming LSA is newer than its current LSDB entry (higher sequence number, or no existing entry).
2. If newer, installs the LSA into the LSDB and floods it out all interfaces except the one it arrived on.
3. If it is a duplicate, sends an acknowledgment but does not re-flood.
4. If the local copy is newer, sends its own copy back to the sender.

This reliable flooding mechanism ensures that a single LSA originated at one router eventually reaches every router in the area. The DR on each broadcast segment plays a central role in flood distribution: DROthers send LSAs to the DR, which re-floods them to the whole segment.

### LSA aging and the age_tick mechanism

Every second, each router increments the age of every LSA record in its LSDB. This is the job of `Lsdb.age_tick`. The function advances time by a given number of seconds and removes any records whose age has reached or exceeded `max_age`.

```
  t=0        t=1800      t=3000      t=3599      t=3600
  LSA born   Age: 1800   Age: 3000   Age: 3599   Age: 3600 → EXPIRED
  installed  (fresh)     (aging)     (near limit) removed from LSDB
                                                  (and flooded with MaxAge
                                                   so neighbors also purge)
```

The originating router prevents expiry by refreshing the LSA before it reaches MaxAge — typically at around the halfway point (1800 seconds). A refresh resets age to 0 and increments the sequence number, restarting the aging clock.

If the originating router fails or becomes unreachable, no refresh arrives. The LSDB age counter for that router's LSAs continues climbing until it reaches `max_age`, at which point `age_tick` removes them. This is the mechanism by which stale topology information is automatically purged from all routers without any explicit deletion message.

### What correct behavior looks like

`Lsdb.age_tick(seconds)` must:

- Increment the `age` field of every `LsaRecord` in `self.records` by `seconds`.
- Identify all records where `age >= self.max_age` after the increment.
- Remove those records from `self.records`.
- Return a list of the `(lsa_id, advertising_router)` tuples for every expired record.

The return value lets the caller know exactly which LSAs were purged so it can trigger the MaxAge flood if needed. The order of entries in the returned list does not need to match any particular ordering, but the set of returned keys must exactly match the set of records removed — no extra entries, no missing ones.

Be careful with mutation order: if you iterate over `self.records` and remove entries during iteration, Python will raise a `RuntimeError`. Collect the keys to remove first, then delete them in a second pass.

### What LSA install and refresh look like (context for the checkpoints)

Although your coding target for this lab is `age_tick`, the scenario also exercises LSA install and refresh paths to set up state for the age-out test. Understanding what these paths do helps you interpret checkpoint failures:

- **Install**: A new `(lsa_id, advertising_router)` key is added to `self.records` with age 0 and the provided sequence number.
- **Refresh**: An existing record's sequence number is incremented and its age is reset to 0. The record is not removed and re-added — it is updated in place.

## Debug trace checkpoints and interpretation guidance

Generate trace artifacts when a step fails:

```bash

```

Checkpoint guide:

- `OSPF_LSA_INSTALL`: Fires when a new LSA record is written into the LSDB for the first time — an `(lsa_id, advertising_router)` key that did not previously exist in `self.records`. The trace details include the lsa_id, advertising_router, and initial sequence number. If this checkpoint is missing, the install path upstream of `age_tick` has a problem; this checkpoint is not produced by your `age_tick` implementation itself, but its absence means the LSDB never has any records to age, which will cause the age-out test to trivially pass with an empty return list rather than actually exercising your code.
- `OSPF_LSA_REFRESH`: Fires when an existing LSA record is updated in place — the sequence number is incremented and the age is reset to 0 for a record that was already present in `self.records`. If this checkpoint is missing, the refresh path is not executing correctly. This may indicate that sequence comparison logic elsewhere in `ospf.py` is rejecting the refresh as a duplicate or stale update. Check that a refresh carries a sequence number strictly greater than the installed copy.
- `OSPF_LSA_AGE_OUT`: Fires when `age_tick` removes one or more records from `self.records` because their age reached `max_age`. The trace details include the list of expired `(lsa_id, advertising_router)` tuples. If this checkpoint is missing after `OSPF_LSA_INSTALL` and `OSPF_LSA_REFRESH` both fired, your `age_tick` implementation is not incrementing age correctly, is using a strict `>` comparison instead of `>=` for the expiry check, or is failing to delete the records from `self.records`. Confirm that you are modifying `self.records` in place and not operating on a temporary copy.
